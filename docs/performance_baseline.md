# Performance Baseline

Дата обновления: 2026-06-02

Performance evidence is generated by:

```bash
make performance-baseline
```

The generated report is written to:

```text
build/release-evidence/PERFORMANCE_BASELINE.md
```

The automated part records frontend `dist` size, largest route chunks and existing binary sizes under the release thresholds. The target OS part must be filled during release smoke on Linux and Windows artifacts with production-like data.

Database query-plan evidence is maintained separately in `docs/db_performance_evidence.md` and validated by:

```bash
make db-performance-check
```

## Budgets

| Metric | Budget |
| --- | --- |
| Startup to login | <= 5 seconds |
| Main list open/search/filter | <= 2 seconds |
| Document registration save | <= 2 seconds typical |
| Heavy statistics/report open | <= 5 seconds |
| Normal desktop memory | <= 512 MB |
| Binary size warning threshold | 100 MB |
| Route chunk warning threshold | 1.6 MB |

## Required Manual Measurements

- startup to login screen;
- login to dashboard ready;
- open main document list;
- search/filter document list;
- save document registration;
- open statistics/report screen;
- memory after normal workflow.
- required PostgreSQL `EXPLAIN (ANALYZE, BUFFERS)` set from `docs/db_performance_evidence.md`.

If target OS timings are not attached for a release candidate, the release owner must explicitly accept the missing evidence in `docs/known_issues.md`.
