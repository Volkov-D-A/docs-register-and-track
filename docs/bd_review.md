# Code Review БД и слоя доступа к данным

Ниже представлен отчет по результатам анализа структуры базы данных и SQL-запросов (слоя доступа к данным) проекта.

## 1. Структура БД: Производительность и Оптимальность

### 1.1. Использование UUID в качестве Primary Key
- **Текущее состояние:** В качестве первичных ключей повсеместно используется `gen_random_uuid()` (UUID v4).
- **Плюсы:** Безопасно для распределенных систем, скрывает реальные объемы данных от пользователей.
- **Минусы/Риски:** Абсолютно случайные UUID приводят к фрагментации B-Tree индексов. При больших объемах вставок это может вызывать повышенную нагрузку на диск (write amplification) и снижение производительности `INSERT`.
- **Рекомендация:** Если в будущем ожидается высокая интенсивность записи, рассмотрите использование последовательных UUID (например, `uuid_generate_v1mc()` из расширения `uuid-ossp`, или генерацию ULID на стороне приложения, сохраняемых как UUID).

### 1.2. Архитектура таблиц (Наследование документов)
- **Текущее состояние:** Применен паттерн Class Table Inheritance: есть базовая таблица `documents` и дочерние таблицы с деталями (например, `incoming_document_details`, `administrative_order_details`), связанные 1-к-1.
- **Плюсы:** Отличная нормализация (DRY). Полиморфные связи (например, поручения, прикрепления файлов) удобно привязывать к общей таблице `documents`.
- **Минусы:** Требуется делать `JOIN` при каждом чтении списка документов конкретного типа.
- **Оценка:** В данном случае это правильный и оптимальный подход, так как выгоды от единой точки входа перевешивают затраты на `JOIN`.

### 1.3. Индексы
- **Текущее состояние:** Хорошо продумано использование внешних ключей и обычных индексов по датам регистрации, типам и связям (например, `idx_documents_registration_date`).
- **Отличная практика:** Использование частичных индексов (Partial Indexes). Пример: `CREATE INDEX idx_admin_order_ack_people_pending ON administrative_order_acknowledgment_people (document_id) WHERE acknowledged_at IS NULL;` — это существенно ускоряет выборки незавершенных действий, экономя память.
- **Риски/Узкие места:** Для текстового поиска (`filter.Search`) используется конструкция `ILIKE '%...%'` по полям `content`, `incoming_number` и т.д. В текущей схеме нет специализированных индексов для этого. БД вынуждена делать `Sequential Scan` (полное сканирование таблицы), что будет очень медленно работать при росте базы.
- **Рекомендация:** Для оптимизации `ILIKE` поиска установите расширение `pg_trgm` и создайте GIN-индексы на поля, по которым идет поиск (например, `CREATE INDEX idx_documents_content_trgm ON documents USING gin (content gin_trgm_ops);`). Альтернатива — использование встроенного полнотекстового поиска PostgreSQL (`tsvector`).

## 2. SQL-запросы и их вызов из приложения

### 2.1. Безопасность (SQL-инъекции)
- **Текущее состояние:** Запросы строятся с использованием позиционных параметров (`$1`, `$2`), а значения передаются через срез аргументов `args`.
- **Оценка:** Сделано абсолютно верно. Приложение защищено от SQL-инъекций.

### 2.2. Проблема N+1 запросов (Критическое замечание)
- **Текущее состояние:** В репозиториях обнаружена классическая проблема N+1 запросов.
  Например, в методе `GetList` репозитория `incoming_doc_repo.go` (и аналогично в `citizen_appeal_repo.go`, `administrative_order_repo.go`):
  ```go
  for rows.Next() {
      ...
      doc.Correspondents, err = r.loadCorrespondents(doc.ID) // <--- Запрос к БД
      resolution, err := r.loadResolution(doc.ID)          // <--- Запрос к БД
      ...
  }
  ```
- **Влияние на производительность:** Если размер страницы (`PageSize`) равен 20, то на один запрос `GetList` выполняется 1 (основной запрос) + 20 + 20 = **41 запрос к базе данных**. Это серьезный `bottleneck`, который может положить базу при увеличении числа одновременных пользователей.
- **Рекомендация:** Загружать связанные сущности батчами.
  - *Вариант А (Простой):* Собрать все `doc.ID` из текущей страницы, затем сделать один запрос `SELECT ... WHERE document_id IN (...)` для корреспондентов и один для резолюций, а затем сгруппировать их в Go-коде с помощью словаря (`map[uuid.UUID][]Entity`).
  - *Вариант Б (Современный PostgreSQL):* Использовать функции агрегации `jsonb_agg` или `json_agg` прямо в основном SELECT-запросе `GetList`, чтобы СУБД сразу отдавала связанные списки в виде JSON-массивов внутри строк документов (Go умеет парсить это напрямую в структуры).

### 2.3. Пагинация (OFFSET vs Keyset)
- **Текущее состояние:** Используется пагинация через `LIMIT $X OFFSET $Y`.
- **Риски:** При глубокой пагинации (например, пользователь хочет посмотреть 1000-ю страницу) БД придется отсканировать и отбросить `1000 * 20` строк. Это ресурсоемко.
- **Рекомендация:** Поскольку в запросах есть жесткая сортировка по `ORDER BY d.created_at DESC`, для критичных таблиц (например, общие журналы или списки всех документов) стоит рассмотреть курсорную пагинацию (Keyset Pagination) через условие `WHERE d.created_at < $cursor`. Для обычных пользовательских интерфейсов `OFFSET` пока приемлем, но стоит иметь это в виду на будущее.

### 2.4. Построение динамических запросов
- **Текущее состояние:** Условия WHERE собираются путем конкатенации строк и слайсов (например, `where = append(where, fmt.Sprintf("d.document_type = $%d", argIdx))`).
- **Оценка:** Подход рабочий, но со временем код репозиториев становится раздутым и сложным для визуального восприятия.
- **Рекомендация:** Для улучшения читаемости можно рассмотреть использование легковесных SQL-билдеров (например, `Masterminds/squirrel` или `doug-martin/goqu`), которые не скрывают SQL-сущность (как ORM), но берут на себя рутину по формированию плейсхолдеров и склейке строк.

### 2.5. Использование вложенных EXISTS в фильтрах
- В запросах `GetList` активно используются конструкции `EXISTS (...) OR EXISTS (...)` для проверки прав доступа к документам (через поручения, ознакомления).
- **Оценка:** СУБД PostgreSQL довольно неплохо оптимизирует `EXISTS`, превращая их в `Semi-Join`. Но при росте таблиц `assignments` и `acknowledgment_users` такие "OR EXISTS" условия могут начать сканировать слишком много данных.
- **Рекомендация:** Рекомендуется периодически мониторить долгие запросы и выполнять `EXPLAIN ANALYZE` на сложных фильтрах. Возможной оптимизацией в будущем станет кеширование прав доступа (деформализация) или использование механизмов RLS (Row-Level Security) на уровне самого PostgreSQL.
