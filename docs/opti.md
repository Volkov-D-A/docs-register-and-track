## 1. Переиспользовать один серверный предикат доступа

Репозитории входящих, исходящих, обращений граждан и распорядительных документов повторяют почти один и тот же SQL-предикат доступа, pagination normalization, count и list. Такое дублирование не только увеличивает кодовую базу: расхождение одного запроса уже создаёт риск утечки данных через вспомогательные экраны.

**Статус на 21 июля 2026 года: реализовано для четырёх основных document lists.** `DocumentQueryService` передаёт явный server-resolved `DocumentAccessScope`, а repository helper добавляет единый параметризованный SQL-предикат и нормализует pagination. Ограниченный scope без номенклатур и subject IDs закрывает выборку (`1=0`), а legacy-поля фильтров временно поддерживаются для прямых вызовов репозиториев. Dashboard, acknowledgments и statistics имеют самостоятельные запросы и не были механически переведены: для них требуется отдельное решение о политике видимости.

Предлагается выделить типизированный компонент уровня repository, например:

```go
type DocumentAccessScope struct {
    Restricted             bool
    AllowedNomenclatureIDs []string
    AccessibleByUserID     string
    AccessibleByUserIDs    []string
}
```

Он добавляет SQL-фрагмент и параметры только через placeholders. Dashboard, acknowledgments, статистика и export следует подключать к тому же scope лишь после явной проверки, что их политика доступа совпадает с document list. Не следует строить SQL из непроверенных строк или превращать builder в универсальный ORM.

Параллельно можно вынести:

- нормализацию `Page`, `PageSize`, sort field и sort direction;
- общий `PageResult[T]`;
- whitelist сортируемых колонок;
- параметризованный contract-test для всех четырёх видов документов.

Ожидаемый результат: уменьшение повторяющегося repository-кода и единая точка исправления правил доступа.

## 2. Устранить N+1-запросы

**Статус на 21 июля 2026 года: N+1 в карточках графа, получателях ознакомлений и pending-выборках устранён.** Graph cards загружаются не более чем одним запросом на вид документа; старые lightweight stores используют совместимый fallback. `GetByDocumentID` и `GetPendingForUser` теперь загружают получателей одним `WHERE acknowledgment_id = ANY($1)`, а pending-задачи для активных замещений — одним запросом по массиву subject IDs. Recursive CTE графа требует отдельной работы: эта оптимизация не меняет его обход и лимиты.

### Граф связей документов

`LinkService.GetDocumentFlow` группирует IDs по виду документа и загружает карточки пакетно через `GetByIDs`: не более одного основного запроса на вид плюс необходимые bulk details. Это заменило отдельную загрузку каждой карточки.

Recursive CTE в `internal/repository/link_repo.go:123-163` включает `depth` в строку `UNION`. Один и тот же edge может повторяться на разных глубинах до финального `DISTINCT`, что создаёт комбинаторный рост на плотном графе. Лучше хранить массив посещённых node/edge IDs, отбрасывать циклы внутри рекурсии и ограничить не только depth, но и максимальное число узлов результата.

### Ознакомления и замещения

N+1 находился в `GetByDocumentID` и `GetPendingForUser`, а не в `GetAllActive`. Получатели теперь загружаются одним запросом `WHERE acknowledgment_id = ANY($1)` и собираются в map.

Pending-задачи для замещений получаются одним `GetPendingForUsers`, который возвращает записи, сгруппированные по subject ID; это сохраняет корректный выбор пользователя при подтверждении ознакомления.

До и после изменения следует считать число SQL-запросов на одну Wails-операцию; это более надёжный acceptance criterion, чем оценка «на глаз».

## 3. Оптимизировать поиск и пагинацию PostgreSQL

Списки ищут через `ILIKE '%строка%'` по содержанию, номерам и наименованиям. Обычные B-tree индексы не помогают при ведущем `%`. После проверки реальных slow queries стоит включить `pg_trgm` и добавить GIN/GiST trigram indexes только на действительно горячие поля. Для больших текстов и сложного ранжирования лучше отдельный `tsvector`, если бизнесу нужен поиск по словам, а не подстроке.

Практический порядок:

1. Включить `pg_stat_statements` и собрать top queries.
2. Зафиксировать планы и фактическое число строк через `EXPLAIN (ANALYZE, BUFFERS)` на production-подобном объёме.
3. Добавить индексы по одному и проверить write overhead/размер.
4. В CI проверять хотя бы корректность миграций, а планы — в performance environment.

Большие `OFFSET` становятся всё дороже. Для основных списков лучше cursor/keyset pagination по стабильной паре `(created_at, id)`. Если UI обязан показывать номер страницы, можно оставить OFFSET для первых страниц и предложить cursor для длинной истории/export.

Отдельные `COUNT(*)` и list-запросы удваивают round trips. В некоторых экранах поможет `COUNT(*) OVER()`, но на пустой глубокой странице общий count не вернётся; решение нужно выбирать по UX и плану запроса, а не применять механически.

## 4. Уменьшить стоимость получения текущего пользователя

`AuthService.GetCurrentUser` вызывает полный `UserRepository.GetByID`, который заодно загружает номенклатуры подразделения и системные permissions. Многие сервисные операции обращаются к текущему пользователю несколько раз.

Следует разделить модели:

- минимальный `SessionPrincipal` — ID, active state, версия отзыва;
- профиль для UI;
- права на конкретный вид документа;
- расширенные справочные данные пользователя.

Logger callback в `internal/app/app.go` получает полный audit profile только ради ID; ему достаточно метода без SQL, возвращающего текущий UUID. Кэшировать разрешения можно только вместе с коротким TTL или версией инвалидирования, иначе оптимизация восстановит проблему неотозванных прав из `bugs.md`.

## 5. Параллелить только независимые операции

Dashboard сначала загружает activity и только затем acknowledgments: `frontend/src/pages/DashboardPage.tsx:50-63`. Эти вызовы независимы и могут выполняться через `Promise.all`, с раздельной обработкой ошибок, если частичный экран допустим.

`StatisticsService` последовательно выполняет несколько независимых агрегатов, например в `internal/services/statistics_service.go:39-54` и `internal/services/statistics_service.go:139-160`. Их можно объединить через `errgroup` и пул PostgreSQL. Однако текущий pool ограничен небольшим числом соединений; параллелизм следует ограничить, иначе один экран статистики будет вытеснять обычные операции.

Правило: сначала замерить последовательную latency и нагрузку, затем параллелить только независимые запросы и поставить per-request concurrency limit.

## 6. Разгрузить frontend bundle

`StatisticsPage` уже загружается лениво на уровне router, но внутри страницы графические библиотеки импортируются верхнеуровнево (`frontend/src/pages/StatisticsPage.tsx:37`). В результате открытие страницы статистики загружает около 1,47 MiB JS плюс крупный chart helper даже если пользователь не открыл вкладку с графиком.

Предлагается:

- вынести вкладки документов, поручений и системы в отдельные lazy components;
- загружать `@ant-design/plots` только при открытии вкладки с графиком;
- использовать прямые imports конкретных chart-компонентов, если пакет гарантирует tree shaking;
- после split повторить bundle analysis и задать CI-budget относительно нового baseline;
- менять chart library только если профилирование покажет, что code splitting недостаточно: миграция ради размера сама по себе может увеличить поддержку.

Крупные route chunks не обязательно являются проблемой, если они редко открываются и хорошо кэшируются; важно отдельно измерять initial route и time-to-interactive статистики.

## 7. Централизовать события и загрузку данных frontend

Dashboard, кнопка пользовательских событий и отдельные страницы могут одновременно реагировать на события и выполнять polling. Без coalescing несколько одинаковых запросов запускаются почти одновременно.

Полезно ввести небольшой слой query cache/store с возможностями:

- один in-flight Promise на одинаковый ключ;
- invalidate по Wails-событию;
- debounce серии событий;
- защита latest-request-wins;
- общий timestamp последней успешной загрузки.

Это не обязательно должен быть крупный сторонний framework. Сначала достаточно общих `useLatestRequest`, `useEventInvalidation` и cache map. Правильный request-ID паттерн уже есть в `frontend/src/hooks/useDocumentListPage.ts:42-78`; его стоит переиспользовать в details, attachments и search links.

В `DashboardPage` локальные компоненты списков определены внутри render (`frontend/src/pages/DashboardPage.tsx:93-217`), поэтому React видит новый тип компонента на каждом render и может перемонтировать subtree. Их следует вынести на уровень модуля и передавать props. Fallback-ключ на `Math.random` (`frontend/src/pages/DashboardPage.tsx:115`) нужно заменить стабильным идентификатором.

## 8. Перейти на потоковую работу с вложениями

Помимо устранения уязвимости это даст самый заметный выигрыш по памяти. Текущий путь создаёт несколько копий файла: browser `File`, data URL/base64 string, decoded `[]byte`, MinIO buffer. Для файла около лимита peak memory кратно превышает его размер, а GC получает большие временные объекты.

Целевая схема:

```text
выбранный локальный путь -> поток/временный файл -> MinIO PutObject
MinIO GetObject -> io.LimitReader -> временный файл/выбранный путь
```

Нужно сохранить серверный контроль размера, MIME не считать доверенным, а checksum вычислять потоково. Для UI можно передавать progress events. После реализации измерить heap profile на файлах 1 MiB, около лимита и больше лимита.

## 9. Не сканировать весь MinIO bucket для каждого экрана статистики

`MinIOStorage.GetStorageInfo` перечисляет все объекты и суммирует размер: `internal/storage/minio.go:96-113`. Стоимость растёт линейно с количеством вложений и блокирует запрос системной статистики.

Варианты, в порядке простоты:

1. TTL-cache результата и singleflight для одновременных запросов.
2. Фоновый сбор метрики с отметкой времени последнего обновления.
3. Поддерживаемый в БД агрегат размера при успешном upload/delete плюс периодическая reconciliation.

Первый вариант минимален, третий быстрее на чтении, но требует надёжной компенсации при расхождении MinIO и PostgreSQL.

## 10. Переиспользование frontend-компонентов

### Формы документов

Формы разных видов повторяют поля регистрации: номенклатура, номер/ручной номер, дата, тип, число листов, содержание. Входящие и обращения граждан дополнительно имеют близкие блоки корреспондентов и резолюций.

Рекомендуется выделить не одну «универсальную форму», а небольшие доменные блоки:

- `RegistrationFields`;
- `DocumentSummaryFields`;
- `CorrespondentListFields`;
- `ResolutionFields` / `ResolutionList`;
- адаптер DTO каждого вида документа.

Так сохраняется типизация и бизнес-специфика, а изменения в общей валидации не копируются по четырём формам. Mega-component с десятками boolean props будет хуже текущего дублирования.

### Фильтры списков

Компоненты в `frontend/src/components/documentFilters` повторяют layout, диапазон дат, поиск и очистку. Существующий `DocumentFilterPanel` можно расширить композиционными primitives:

- `FilterRow`;
- `DateRangeFilter`;
- `SearchField` с debounce;
- `ClearFiltersButton`;
- slots для kind-specific фильтров.

### CRUD справочников

Вкладки организаций и исполнителей резолюций в settings реализуют близкие table/modal/create/update/delete сценарии. Типизированные `useDirectoryCrud<TForm, TRow>` и `DirectoryTable` сократят обработчики и унифицируют loading/error/confirm. Колонки и форма должны передаваться конфигурацией, а permissions оставаться снаружи.

### Async hooks

Общий `useLatestRequest` должен инкапсулировать sequence ID, mounted-state и раздельные `data/error/loading`. Это сократит код и устранит гонки в `useDocumentDetails`, `useAttachments` и `LinksTab`.

## 11. Упростить backend без потери доменной ясности

- Разделить крупный mapper DTO по bounded contexts. Один файл на 1000+ строк сложнее ревьюить и тестировать; reflection ради сокращения строк применять не стоит.
- Сохранить общую транзакционную основу регистрации документов, но не объединять разные виды в один сервис с ветвлением по string kind.
- Удалить дублирующие комментарии в `internal/services/link_service.go:107-110` и прочие механические повторы отдельным безопасным cleanup commit.
- Generated Wails bindings не считать ручным дублированием и не редактировать вручную; оптимизировать нужно исходные DTO/API.
- Ввести небольшие helpers для `NotFound`, `RowsAffected` и нормализации pagination, чтобы одинаковые ошибки не исправлялись в каждом репозитории отдельно.

## 12. Таймауты и наблюдаемость как часть оптимизации

Без измерений ускорение легко превратить в перегрузку БД. Рекомендуемый минимальный набор:

- p50/p95/p99 времени каждого Wails method;
- число SQL-запросов и DB time на operation;
- pool wait time и занятые соединения;
- размер result set и число узлов link graph;
- heap/allocations при upload/download;
- размер route chunks и время загрузки;
- счётчики outbox/retry/reconciliation для MinIO и аудита.

Все SQL-операции должны принимать context и иметь разумный deadline. Медленные отчёты получают отдельный, больший лимит и не должны занимать весь pool.

## Критерии приёмки оптимизаций

Конкретные миллисекунды нужно согласовать по production-профилю, но обязательны относительные проверки:

- количество SQL-запросов графа и активных ознакомлений не растёт линейно с числом строк;
- глубокая страница не требует сканирования всех предыдущих строк;
- открытие dashboard не загружает chart library;
- повторные одинаковые события не создают параллельные одинаковые запросы;
- peak memory при вложениях растёт как небольшой buffer, а не как несколько полных копий файла;
- каждый общий компонент имеет contract-тесты для всех использующих его видов документа;
- bundle и latency budgets закреплены в CI после фиксации нового baseline.
