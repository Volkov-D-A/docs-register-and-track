# Ревью обработки ошибок проекта

## 1. Общая архитектура и текущие подходы

В проекте реализован современный и структурированный подход к обработке ошибок:
- **Базовая структура (`internal/models/errors.go`)**: Используется кастомный тип `AppError`, который содержит HTTP-подобный код статуса (`Code`), строковый идентификатор ошибки (`Kind`), человекочитаемое сообщение (`Message`) и оригинальную внутреннюю ошибку (`Internal`). Присутствуют удобные конструкторы: `NewBadRequest`, `NewUnauthorized`, `NewInternal` и т.д.
- **Уровень доступа к данным (Repository)**: Репозитории используют стандартное для Go оборачивание ошибок через `fmt.Errorf("...: %w", err)`. Это правильный (idiomatic Go) паттерн, позволяющий сохранять исходную ошибку для логирования и проверок через `errors.Is`/`errors.As`.
- **Управление транзакциями**: В репозиториях повсеместно применяется паттерн `defer tx.Rollback()`. Это безопасно, так как после успешного `tx.Commit()` вызов `Rollback()` просто возвращает ошибку `sql: transaction has already been committed`, которая корректно игнорируется.
- **Связующий слой Wails (`main.go`)**: В `options.App` настроен `ErrorFormatter`. Он перехватывает все ошибки, возвращаемые из Go-методов (Bindings). Если ошибка является `AppError`, она пробрасывается на фронтенд с сохранением кода и сообщения. Если это любая другая ошибка (например, созданная через `fmt.Errorf`), она подменяется на абстрактную `INTERNAL_ERROR` (500) с сообщением "произошла внутренняя ошибка". Оригинальная ошибка при этом пишется в лог.
- **Фронтенд (`frontend/src/utils/appError.ts`)**: Присутствует развитая система маппинга кодов ошибок (CODE_COPY) на пользовательские сообщения и подсказки по дальнейшим действиям. Ошибки с бэкенда корректно парсятся и обрабатываются.

## 2. Выявленные проблемы и неоптимальности

Главная проблема кроется на **Сервисном уровне (Services)** — в месте формирования бизнес-ошибок.

### Проглатывание ошибок валидации (Скрытые 500-е ошибки)
Из-за строгой логики `ErrorFormatter` в `main.go`, любая ошибка, не являющаяся `AppError`, превращается в "произошла внутренняя ошибка".
Однако в сервисных хендлерах (например, `citizen_appeal_command_handler.go`, `administrative_order_command_handler.go`, `document_access_admin_service.go`) часто используется `fmt.Errorf` для возврата бизнес-ошибок и ошибок валидации, которые **должен** увидеть пользователь.

**Примеры из кода:**
- В `citizen_appeal_command_handler.go`:
  ```go
  parsed, err := time.Parse("2006-01-02", strings.TrimSpace(value))
  if err != nil {
      // Ошибка парсинга даты вернётся как fmt.Errorf.
      // Пользователь увидит "внутренняя ошибка", а не "неверный формат даты".
      return time.Time{}, fmt.Errorf("неверный формат %s: %w", fieldName, err)
  }
  ```
  Или при поиске исполнителя:
  ```go
  if _, err := h.refRepo.FindOrCreateResolutionExecutor(name); err != nil {
      // То же самое: сообщение будет скрыто от пользователя.
      return fmt.Errorf("ошибка исполнителя резолюции: %w", err)
  }
  ```
- В `document_access_admin_service.go`:
  ```go
  return fmt.Errorf("действие %q не поддерживается для вида документа %q", permission.Action, permission.KindCode)
  ```

Такая неоднородность приводит к тому, что часть полезных сообщений о неверно заполненных полях теряется по пути на фронтенд.

## 3. Рекомендации по улучшению

1. **Единообразие в Сервисном слое**: 
   Все ошибки валидации, нарушений бизнес-логики или инвариантов, текст которых предназначен для конечного пользователя, **обязательно** должны возвращаться через конструкторы `models.AppError` (например, `models.NewBadRequest(fmt.Sprintf(...))`). Использование `fmt.Errorf` или `errors.New` на уровне сервисов должно быть зарезервировано исключительно для непредвиденных системных сбоев (например, отвалилась БД), которые действительно должны отображаться как `INTERNAL_ERROR`.

2. **Оборачивание в AppError**:
   Если нужно вернуть пользователю сообщение, но при этом сохранить для логов оригинальную ошибку (как это делает `%w`), рекомендуется доработать `models.AppError`. Сейчас `NewBadRequest` принимает только `msg string`. 
   *Решение:* Можно добавить конструктор вроде `models.NewBadRequestWrapped(msg string, err error)`, который будет инициализировать поле `Internal`. Таким образом `ErrorFormatter` в `main.go` сможет отдать `msg` пользователю, а в лог записать детали из `Internal`.

3. **Провести аудит `fmt.Errorf` в папке `internal/services`**:
   Рекомендуется пройтись поиском по `fmt.Errorf` в сервисах и заменить те вызовы, которые описывают ошибки пользовательского ввода или бизнес-логики, на `models.NewBadRequest`.

4. **Фронтенд и `allowDetail`**:
   В `appError.ts` для некоторых кодов ошибок (например, `INTERNAL_ERROR`) отключен флаг `allowDetail`. Это правильно с точки зрения безопасности (чтобы не показывать stacktrace), но если бэкенд научится разделять системные и пользовательские ошибки строго по `AppError`, фронтенд будет получать только "безопасные" сообщения.

**Итог:** Архитектура обработки ошибок выбрана хорошая и расширяемая, но требует строгого соблюдения контракта (всегда возвращать `models.AppError` для клиентских ошибок) на уровне бизнес-логики.
