# Migration Rollback Runbook

Дата: 2026-05-28

Этот runbook применяется перед откатом последней миграции через экран настроек приложения.

## Preconditions

- Все пользователи завершили работу в приложении.
- Создана свежая резервная копия PostgreSQL и MinIO одним согласованным backup workflow.
- Оператор записал идентификатор или путь к архиву backup.
- Оператор понимает, что `down`-миграция может удалить таблицы, столбцы и данные.

## Rollback Procedure

1. Открыть `Настройки -> Миграции`.
2. Проверить текущую версию схемы и отсутствие `dirty` состояния.
3. Нажать `Откатить последнюю`.
4. В форме подтверждения отметить наличие свежего backup PostgreSQL+MinIO.
5. Указать идентификатор или путь к backup archive.
6. Подтвердить риск потери данных.
7. Ввести контрольную фразу `ОТКАТ МИГРАЦИИ`.
8. Запустить откат.

Backend отклоняет rollback без всех подтверждений. Перед выполнением отката пишется audit entry `MIGRATION_ROLLBACK_REQUESTED`, после успешного отката пишется `MIGRATION_ROLLBACK`.

## Post-Rollback Checks

- Повторно открыть статус миграций и убедиться, что версия уменьшилась на один шаг и `dirty=false`.
- Проверить вход администратора.
- Проверить открытие списков документов, журналов и настроек.
- Проверить, что ожидаемые данные после отката соответствуют impact review для конкретной `down`-миграции.
- Если откат завершился ошибкой или schema стала `dirty`, остановить работу с приложением и выполнять восстановление по backup/restore runbook.

## Release Evidence

Перед релизом выполнить rollback policy test на disposable PostgreSQL database:

- попытка rollback без backup confirmation должна завершиться `VALIDATION_ERROR`;
- попытка rollback без контрольной фразы должна завершиться `VALIDATION_ERROR`;
- rollback с полным подтверждением должен создать audit entry и изменить версию схемы только на тестовой БД;
- после rollback выполнить smoke checks и при необходимости restore test из свежего backup.
