### 1. High: журнал документа доступен любому аутентифицированному пользователю

Статус:
- исправлено

Файл: `internal/services/journal_service.go`

Проблема:
- метод `GetByDocumentID` проверяет только факт аутентификации;
- проверки доступа к самому документу нет;
- в отличие от файлов и карточек документа, здесь не используется `requireDocumentReadAccess` или `requireAnyDocumentReadAccess`.

Риск:
- исполнитель, знающий UUID документа, может получить журнал чужого документа через Wails binding;
- это раскрывает служебную историю действий по документу.

Место в коде:
- `internal/services/journal_service.go:26`

Рекомендация:
- добавить такую же проверку доступа к документу, как в сервисах документов и вложений;
- в идеале переиспользовать уже существующие policy-хелперы.

Что сделано:
- добавлен `DocumentAccessService` как единая точка проверки доступа к документному домену;
- `JournalService` переведен на проверку доступа к конкретному документу перед выдачей журнала;
- та же точка проверки переиспользована в сервисах документов и вложений для дальнейшего масштабирования.

Файлы исправления:
- `internal/services/document_access_service.go`
- `internal/services/journal_service.go`
- `internal/services/incoming_doc_service.go`
- `internal/services/outgoing_doc_service.go`
- `internal/services/attachment.go`
- `main.go`

### 2. Medium: поручения и ознакомления можно создать для несуществующего документа

Файлы:
- `internal/services/assignment_service.go`
- `internal/services/acknowledgment_service.go`
- `internal/database/migrations/004_assignments.up.sql`
- `internal/database/migrations/005_attachments_and_settings.up.sql`

Проблема:
- сервис создания поручения принимает `documentID` и `documentType`, но не проверяет существование документа;
- сервис создания ознакомления делает то же самое;
- схема БД не содержит FK на документы и не ограничивает `document_type` допустимыми значениями.

Риск:
- можно создать "висячие" поручения и ознакомления;
- такие записи потом создают расхождения в списках, графах связей и журнале;
- данные становятся логически неконсистентными, хотя физически запись успешно сохраняется.

Места в коде:
- `internal/services/assignment_service.go:38`
- `internal/services/acknowledgment_service.go:38`
- `internal/database/migrations/004_assignments.up.sql:2`
- `internal/database/migrations/005_attachments_and_settings.up.sql:44`

Рекомендация:
- валидировать `documentType` на уровне сервиса;
- перед созданием проверять существование документа в соответствующем репозитории;
- по возможности усилить ограничения схемы хотя бы `CHECK` для `document_type`.

### 3. Medium: возможен `nil`-dereference в `GetCurrentAuditInfo`

Файл: `internal/services/auth_service.go`

Проблема:
- метод `GetCurrentAuditInfo` обрабатывает ошибку репозитория;
- но не обрабатывает сценарий, когда `userRepo.GetByID` вернул `nil, nil`;
- затем код обращается к `user.ID` и `user.FullName`.

Риск:
- если пользователь был удален из БД после логина, сервис может упасть на побочном пути аудита/логирования;
- такие падения особенно неприятны из-за слабой воспроизводимости.

Место в коде:
- `internal/services/auth_service.go:303`

Рекомендация:
- добавить явную проверку `if user == nil`;
- в таком случае возвращать безопасное значение вроде `userID, "system"` или аналогичный fallback.

### 4. Low: фронтенд оптимистично меняет активную роль без отката

Файл: `frontend/src/store/useAuthStore.ts`

Проблема:
- `currentRole` меняется в store до завершения `SetActiveRole`;
- если backend отклонит смену роли, UI не откатывает состояние;
- ошибка только логируется в консоль.

Риск:
- интерфейс может начать отображать неподходящие экраны и действия;
- затем пользователь будет получать `Forbidden` уже на последующих запросах.

Место в коде:
- `frontend/src/store/useAuthStore.ts:154`

Рекомендация:
- обновлять `currentRole` только после успешного ответа backend;
- либо делать optimistic update с явным rollback в `catch`.

## Дополнительные наблюдения

### Использование `documentType` при создании поручений

В сервисе создания поручений используются оба поля:
- `documentID`
- `documentType`

Это ожидаемая модель, потому что поручения могут относиться как к входящему, так и к исходящему документу.

При этом сейчас `documentType` используется без строгой валидации, что и приводит к одному из замечаний выше.
