# Production-аудит проекта Wails + React + Ant Design + PostgreSQL

Версия: 2.0  
Формат: один рабочий промпт на каждый этап + развернутая нумерация пунктов.

## Как пользоваться этим планом

Этот план рассчитан на последовательную работу с ИИ. Не нужно отправлять весь план сразу. Выбирай этап или конкретный диапазон пунктов и передавай ИИ соответствующий промпт этапа.

Примеры команд:

```text
Сейчас занимаемся этапом B полностью.
```

```text
Сейчас занимаемся только пунктами B.04.053–B.04.064: индексы PostgreSQL.
```

```text
Сейчас занимаемся этапом D, но только подпунктом D.05: формы Ant Design и валидация.
```

```text
Вернись к пунктам C.03.086–C.03.090 и перепроверь их после изменений DTO.
```

Каждый пункт имеет код вида:

```text
B.04.053
```

Где:

```text
B   — этап;
04  — тематический блок внутри этапа;
053 — глобальный номер пункта из исходного плана.
```

Глобальная нумерация 001–270 сохранена, чтобы можно было отслеживать прогресс и регрессии.

# 1. Общие правила ведения аудита

## 1.1. Что хранить в `PROJECT_CONTEXT.md`

`PROJECT_CONTEXT.md` — главный постоянный контекст. Его нужно давать ИИ почти на каждом этапе.

В него попадает только устойчивое:

```text
- назначение приложения;
- основные сущности;
- бизнес-инварианты;
- роли и разрешения;
- границы frontend/backend/database;
- production-ограничения;
- терминология;
- архитектурные решения;
- риски, влияющие на следующие этапы.
```

В него не нужно складывать:

```text
- все найденные баги;
- временные наблюдения;
- уже исправленные проблемы;
- длинные SQL-планы;
- полные логи сборки;
- косметические замечания;
- неподтвержденные догадки.
```

## 1.2. Формат записи в `REVIEW_LOG.md`

```text
## ISSUE-001

Категория: Database
Пункт плана: B.04.053
Severity: critical / major / minor
Статус: open / fixed / accepted / postponed / false_positive
Место: ...
Проблема: ...
Почему важно: ...
Рекомендация: ...
Проверка после исправления: ...
Связанные пункты: ...
```

## 1.3. Формат записи в `DECISIONS.md`

```text
## DECISION-001

Дата: ...
Контекст: ...
Решение: ...
Причина: ...
Альтернативы: ...
Последствия: ...
Какие этапы затрагивает: ...
```

## 1.4. Формат записи в `RISK_REGISTER.md`

```text
## RISK-001

Категория: Backend
Описание: ...
Вероятность: low / medium / high
Влияние: low / medium / high
Митигирующее действие: ...

```

## 1.5. Формат записи в `REGRESSION_MAP.md`

```text

Что изменено: ...
Затронутые файлы: ...
Затронутые пункты плана: ...
Что перепроверить: ...
Что не нужно перепроверять: ...
```

## 1.6. Универсальный формат отчета по пункту

Используй этот формат для любого пункта плана:

```text
## Пункт B.04.053

Статус: ok / issue / not_applicable / needs_info
Severity: critical / major / minor / none
Место: файл / модуль / таблица / экран / сценарий
Доказательство: ...
Проблема: ...
Почему важно: ...

Риск исправления: ...
Что перепроверить после исправления: ...
Связанные пункты: ...
```
