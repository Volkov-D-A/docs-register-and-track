## 1. Деактивация пользователя не отзывает активную сессию

**Приоритет:** высокий, нарушение контроля доступа.

`AuthService` хранит только `currentUserID`. Метод `IsAuthenticated` проверяет, что идентификатор не `Nil`, а `GetCurrentUser` загружает пользователя, но не проверяет `IsActive`: `internal/services/auth_service.go:31-38`, `internal/services/auth_service.go:207-224`, `internal/services/auth_service.go:325-328`. Проверки прав через `RequireSystemPermission` также опираются на этот признак: `internal/services/auth_service.go:381-388`.

При блокировке после неудачных входов или при административном отключении репозиторий выставляет `is_active = false`: `internal/repository/user_repo.go:218-231`, `internal/repository/user_repo.go:450-465`. Уже открытый экземпляр приложения об этом не узнаёт и продолжает выполнять операции от имени отключённого пользователя, пока тот сам не выйдет или процесс не перезапустится.

**Как проверить:** войти одним пользователем в экземпляре A, отключить его администратором в экземпляре B, затем выполнить разрешённую операцию в A.

**Исправление:** единообразно получать активного principal перед каждой защищённой операцией; при `is_active = false` очищать локальную сессию и возвращать `ErrUnauthorized`. Для немедленного отзыва также можно хранить в БД версию/эпоху авторизации и сверять её с версией сессии. Нельзя кэшировать права без механизма инвалидирования.

**Регресс-тест:** параметризованный тест всех permission gates: активный пользователь проходит, отключённый после входа получает отказ без перезапуска приложения.

## 2. Dashboard раскрывает поручения из недоступных видов документов

**Приоритет:** высокий, утечка конфиденциальных данных.

В `DashboardService.GetExpiringAssignments` наличие действия `create` или `read` хотя бы для одного вида документа превращается в глобальный доступ: `internal/services/dashboard_service.go:44-58`. После этого репозиторий вызывается без ограничения по пользователю: `internal/services/dashboard_service.go:84-91`.

Запрос `DashboardRepository.GetExpiringAssignments` не фильтрует вид документа и область чтения, но возвращает номер, содержание, исполнителя и сроки: `internal/repository/dashboard_repo.go:44-57`. В результате пользователь с правом чтения одного вида документов может видеть поручения, относящиеся к другим, недоступным видам.

**Исправление:** передавать в запрос фактическую область доступа: разрешённые виды документов плюс правила доступа к конкретному документу. Использовать ту же модель предикатов, что и списки документов, а не сводить «есть право хотя бы где-то» к глобальному scope.

**Регресс-тест:** матрица из четырёх видов документов и ролей; dashboard не должен возвращать строки, для которых отдельный `GetByID` запрещён.

## 3. `GetAllActive` ознакомлений раскрывает все документы и получателей

**Приоритет:** высокий, утечка данных.

`AcknowledgmentService.GetAllActive` проверяет лишь наличие права `acknowledge` для любого вида: `internal/services/acknowledgment_service.go:253-264`. Затем `AcknowledgmentRepository.GetAllActive` выбирает все активные записи без фильтра по виду/области чтения и включает вид, номер, содержание и автора документа: `internal/repository/acknowledgment_repo.go:339-382`. Для каждой записи отдельно загружается список пользователей: `internal/repository/acknowledgment_repo.go:371-375`.

Пользователь с правом ознакомления для одного вида получает данные активных ознакомлений других видов и сведения об их участниках.

**Исправление:** ограничить выборку разрешёнными видами и реальной областью чтения документа. Если экрану нужны только агрегаты, возвращать минимальный DTO без содержания и списка пользователей. Добавить серверный тест, который сравнивает результаты реестра с результатами проверки доступа к каждому документу.

## 4. Исполнитель может вернуть финализированное поручение в работу

**Приоритет:** высокий, нарушение целостности процесса.

`resolveAssignmentStatusUpdate` разрешает исполнителю целевые состояния `in_progress` и `completed`, не проверяя текущее состояние: `internal/services/assignment_service.go:97-134`. Метод `UpdateStatus` применяет это решение напрямую: `internal/services/assignment_service.go:246-292`.

Поэтому исполнитель поручения со статусом `finished` может установить `in_progress` или `completed`. Это противоречит остальному коду, где `finished` явно считается неизменяемым при редактировании и удалении: `internal/services/assignment_service.go:213-215`, `internal/services/assignment_service.go:597-600`.

**Исправление:** заменить набор разрозненных условий явной таблицей переходов `(текущий статус, целевой статус, роль)`. `finished` и `cancelled` должны быть терминальными, если бизнес-требования не предусматривают отдельную административную операцию повторного открытия.

**Регресс-тест:** табличный тест всех пар состояний и ролей; особенно `finished -> in_progress/completed` и `cancelled -> *`.

## 5. Возможно отключить или лишить прав последнего администратора

**Приоритет:** высокий, административный отказ в обслуживании.

`UserService.UpdateUser` позволяет администратору отключить любого пользователя, включая себя или последнего активного администратора: `internal/services/user_service.go:67-79`. `DocumentAccessAdminService.UpdateUserAccessProfile` аналогично позволяет убрать административное право у самого себя или последнего администратора: `internal/services/document_access_admin_service.go:38-66`.

Повторная первичная настройка не поможет: `InitialSetup` запрещён, как только в БД уже есть хотя бы один пользователь: `internal/services/auth_service.go:441-447`.

**Исправление:** в одной транзакции проверять инвариант «существует хотя бы один активный пользователь с административным правом». Запретить деактивацию/удаление последнего администратора и снятие его последнего административного права. Проверка только в UI недостаточна — она должна быть на сервере, желательно подкреплена ограничением или блокировкой строк в БД.

## 6. Первичная настройка неатомарна

**Приоритет:** высокий, система может остаться без администратора.

`InitialSetup` сначала создаёт пользователя, а затем отдельной операцией назначает ему профиль администратора: `internal/services/auth_service.go:428-476`. Создание пользователя фиксируется репозиторием отдельно. Если `ReplaceUserAccessProfile` завершится ошибкой, пользователь уже существует, но прав у него нет; следующая попытка блокируется проверкой количества пользователей.

Параллельные вызовы первичной настройки также не защищены общей транзакцией/блокировкой.

**Исправление:** выполнять проверку, создание пользователя и выдачу административного права в одной транзакции. Дополнительно использовать advisory lock или другой межпроцессный mutex в PostgreSQL. Процедуру восстановления следует сделать идемпотентной: если пользователи есть, но активного администратора нет, должен существовать контролируемый аварийный путь восстановления.

**Тест:** fault injection на этапе назначения прав должен оставлять БД без нового пользователя либо с полностью настроенным администратором, но не в промежуточном состоянии.

## 7. Сброс пароля несуществующего UUID приводит к panic

**Приоритет:** средний.

`UserService.ResetPassword` получает пользователя, но не проверяет результат на `nil`, а затем обращается к `user.FullName`: `internal/services/user_service.go:88-108`. `UserRepository.UpdatePassword` не проверяет `RowsAffected`, поэтому корректный, но отсутствующий UUID не превращается в `NotFound`: `internal/repository/user_repo.go:405-421`.

Итог — nil pointer dereference в сервисном методе. Конкретный масштаб отказа зависит от механизма восстановления panic в Wails, но текущая UI-операция точно не получает нормальную доменную ошибку.

**Исправление:** проверять существование пользователя до обновления, а в репозитории превращать `RowsAffected == 0` в `ErrNotFound`. Добавить recovery на границе binding как последнюю линию защиты, но не использовать его вместо исправления причины.

## 8. Создание связи и отмена приказа неатомарны

**Приоритет:** высокий, нарушение целостности.

`LinkService.Create` сначала сохраняет связь, затем отдельным запросом отменяет связанный приказ: `internal/services/link_service.go:98-105`. Если отмена не удалась, связь остаётся созданной, клиент получает ошибку, а повтор запроса может упасть на уникальности уже существующей связи. Состояние приказа при этом не соответствует смыслу связи.

**Исправление:** объединить создание связи и изменение статуса приказа в одну SQL-транзакцию. Репозитории должны принимать общий `tx`. Уведомления и внешние эффекты выполнять после commit или через transactional outbox.

## 9. Удаление вложения может оставить битую запись БД

**Статус: исправлено 16 июля 2026 года.** Удаление выполняется через `attachment_delete` outbox/saga: запись сначала скрывается в БД, а worker повторяемо удаляет объект MinIO и затем marked-строку. Добавлена административная read-only сверка БД и bucket.

**Приоритет:** высокий, потеря доступности документа.

В одиночном удалении объект сначала удаляется из MinIO, и только затем удаляется запись БД: `internal/services/attachment.go:231-239`. Массовое удаление повторяет тот же порядок: `internal/services/attachment.go:469-485`. Ошибка БД после успешного удаления объекта оставляет видимое вложение, скачать которое уже невозможно.

Обычная транзакция не может атомарно охватить PostgreSQL и MinIO.

**Исправление:** использовать saga/outbox: в транзакции помечать запись как `deleting`, после commit удалять объект, затем финализировать удаление. Неуспешные шаги должны повторяться фоновым reconciler. Нужны метрики и административная команда сверки БД с bucket.

## 10. Обработка вложений допускает неконтролируемое потребление памяти

**Приоритет:** высокий, отказ в обслуживании.

Frontend читает файл целиком через `FileReader.readAsDataURL`, создавая base64-копию: `frontend/src/hooks/useAttachments.tsx:43-63`. Backend затем декодирует всю строку и только после выделения памяти проверяет лимит: `internal/services/attachment.go:83-96`. Вызов Wails binding можно сделать напрямую из скомпрометированного renderer, поэтому UI-лимит не является защитой.

На скачивании `MinIOStorage.Download` использует неограниченный `io.ReadAll`: `internal/storage/minio.go:72-84`. Затем данные снова копируются при base64-кодировании или сохранении: `internal/services/attachment.go:196-205`, `internal/services/attachment.go:278-300`. Ошибочный или подменённый крупный объект может исчерпать память процесса.

**Исправление:** перед декодированием проверять максимальную допустимую длину base64, но основное решение — потоковая загрузка из файла без JS/base64. При скачивании сначала проверять `StatObject`, читать через `io.LimitReader(max+1)` и потоково писать во временный файл. Лимит должен проверяться сервером независимо от метаданных клиента.

## 11. Секреты попадают в аргументы процессов, журналы сборки и бинарник

**Приоритет:** высокий, раскрытие секретов.

CLI-параметр `--encrypt-password` принимает пароль открытым аргументом: `main.go:28`. Он может сохраняться в истории shell и быть виден в списке процессов.

`Makefile` передаёт `ENCRYPTION_KEY` через linker flags: `Makefile:15-16`, `Makefile:30`, `Makefile:36`, `Makefile:42`. Команда может попасть в CI-log и список процессов, а сам ключ оказывается внутри распространяемого бинарника. Документация прямо описывает такую схему: `docs/tech_docs.md:259`. Пользователь, имеющий бинарник, в принципе может извлечь встроенный секрет, поэтому это не является надёжной защитой конфигурации.

**Исправление:** пароль для helper-команды читать из защищённого интерактивного ввода или stdin, не из argv. Ключ приложения получать при запуске из OS credential store, защищённого файла/секрет-хранилища или дескриптора, не печатать в build-команде и не встраивать в бинарник. План ротации ключа и повторного шифрования конфигурации должен быть документирован.

## 12. Backup/restore-скрипты исполняют `.env` как root-код

**Приоритет:** высокий, локальное повышение привилегий.

`backup_smb_tar.sh` и `restore_smb_tar.sh` делают `source` жёстко заданного project `.env`: `backup_smb_tar.sh:6-14`, `restore_smb_tar.sh:7-15`. Восстановление предполагается запускать через `sudo`: `restore_smb_tar.sh:24`; монтирование также требует повышенных прав.

`.env` — это не формат данных, а валидный shell-скрипт. Если файл или один из его родительских каталогов доступен на запись непривилегированному пользователю, тот может выполнить произвольную команду от root. Это также противоречит рекомендации `.envExample` хранить production-конфигурацию вне Git с режимом `600`.

Кроме того, пароль MinIO подставляется внутрь `sh -c`: `backup_smb_tar.sh:92-93`, `restore_smb_tar.sh:167-170`. Он может быть виден в argv/container metadata, а специальные shell-символы способны изменить исполняемую команду.

**Исправление:** читать конфигурацию строгим parser формата `KEY=VALUE`, а не `source`; путь сделать настраиваемым и указывать на root-owned файл вне рабочей копии. Перед чтением проверять владельца, права и отсутствие symlink. Привилегии повышать только на минимальный шаг монтирования, а пароль передавать через stdin/secret file без `sh -c`.

## 13. Подтверждение ознакомления неидемпотентно

**Статус: исправлено 16 июля 2026 года.** Подтверждение защищено предикатом `confirmed_at IS NULL`; повтор считается успешным без повторного журнала или уведомлений. Обязательные эффекты фиксируются в outbox в общей транзакции.

**Приоритет:** средний, искажение аудита и лишние уведомления.

UPDATE получателя не содержит условия `confirmed_at IS NULL`: `internal/repository/acknowledgment_repo.go:294-309`. Повторный вызов снова меняет время подтверждения и повторно обновляет `completed_at` родителя: `internal/repository/acknowledgment_repo.go:323-330`. Сервис при каждом таком успехе пишет журнал и отправляет события: `internal/services/acknowledgment_service.go:310-326`.

**Исправление:** добавить `confirmed_at IS NULL`, различать первый переход и уже выполненное подтверждение и эмитить событие только один раз. Для уведомлений полезен outbox с уникальным ключом бизнес-события.

## 14. Ошибки записи аудита игнорируются

**Статус: исправлено 16 июля 2026 года.** Journal и административный аудит доставляются через outbox с retry, terminal failure, мониторингом и ручным повтором. Состояние и повтор terminal-задач доступны администраторам через `OutboxAdminService`.

**Приоритет:** средний, риск для расследований и соответствия требованиям.

`AdminAuditLogService` явно отбрасывает ошибку `repo.Create`: `internal/services/admin_audit_log_service.go:52-61`. Во многих сервисах аналогично игнорируется результат `journal.LogAction`, например `internal/services/attachment.go:145-150`, `internal/services/attachment.go:242-247` и операции поручений в `internal/services/assignment_service.go:172-181`, `internal/services/assignment_service.go:277-287`.

Таким образом, бизнес-операция может успешно завершиться без аудиторского следа, а приложение и оператор об этом не узнают.

**Исправление:** зафиксировать политику аудита. Для обязательного аудита писать событие в той же транзакции или transactional outbox. Для best-effort аудита хотя бы возвращать/логировать ошибку, ставить retry и метрику. Нельзя молча отбрасывать ошибку на уровне сервиса.

## 15. Подключение и большинство SQL-запросов не имеют таймаута

**Статус: исправлено 19 июля 2026 года.** Подключение проверяется через `PingContext` с коротким startup-timeout. Общая обёртка БД ограничивает ожидание пула и длительность legacy `Query`/`Exec`/транзакций, сохраняя возможность передать более короткий context в новых операциях. На уровне PostgreSQL для всех соединений настроены `statement_timeout` и `lock_timeout`.

**Приоритет:** средний, зависание приложения при сетевых проблемах.

При запуске используется `db.Ping()` без контекста: `internal/database/postgres.go:64-74`. При недоступной сети он может надолго блокировать старт. Большинство репозиториев использует `Query`/`Exec`, а не `QueryContext`/`ExecContext`; жизненный цикл Wails-операции до SQL не доходит.

**Исправление:** использовать `PingContext` с коротким startup-timeout; передавать request context через сервисы и репозитории; настроить `statement_timeout`, `lock_timeout` и отдельно таймауты для долгих отчётов/миграций. Отмена UI-операции должна отменять соответствующий запрос к БД.

## 16. Устаревшие async-ответы перезаписывают актуальное состояние frontend

**Статус: исправлено 19 июля 2026 года.** Загрузка документа, вложений, списка связей и поиск документов используют общий sequence guard. Смена документа, вида или поисковой строки инвалидирует выполняющиеся запросы; их устаревшие success, error и finally не изменяют состояние нового экрана.

**Приоритет:** средний, функциональная ошибка и риск выбора неверного документа.

`useDocumentDetails` запускает запрос при смене `documentId`, но не использует sequence token или отмену: `frontend/src/hooks/useDocumentDetails.ts:14-24`. Старый ответ способен установить данные после открытия нового документа; старый error может вызвать callback, который закрывает уже новый modal: `frontend/src/components/DocumentViewModal.tsx:46-55`.

Та же гонка есть при загрузке вложений: `frontend/src/hooks/useAttachments.tsx:23-35`. В поиске связей старый ответ для предыдущего `targetKind`/текста может заменить новые варианты: `frontend/src/components/DocumentLinks/LinksTab.tsx:92-142`.

В проекте уже есть корректный паттерн с `requestId` в `frontend/src/hooks/useDocumentListPage.ts:42-78`; его следует вынести и переиспользовать.

**Тест:** вернуть первый Promise позже второго и проверить, что UI показывает только результат последнего запроса и не вызывает старый `onError`.

## Дополнительные замечания низкого приоритета

- Политика пароля измеряет `len(password)` в байтах, а не количество Unicode-символов: `internal/security/password.go:17-45`. Если минимальная длина трактуется как число символов, следует использовать `utf8.RuneCountInString` и явно определить допустимый алфавит.
- Блокировка после пяти неудачных входов постоянна и не имеет временного окна/задержки: `internal/services/auth_service.go:88-99`, `internal/repository/user_repo.go:450-465`. Это позволяет локально заблокировать известную учётную запись. Лучше использовать возрастающую задержку, временную блокировку, журналирование и защищённое административное восстановление.
- При отсутствии ключа шифрования вызывается `panic`: `internal/config/crypto.go:28-42`. Ошибки расшифровки в части getters фактически превращают `ENC(...)` в невалидное значение конфигурации. Старт приложения должен завершаться структурированной диагностикой без вывода секрета.
- **Исправлено 19 июля 2026 года:** backup публикуется атомарно через временные файлы на том же SMB mount; после flush и проверки архива вычисляются размер и SHA-256, а manifest публикуется последним как commit marker. Restore до распаковки требует и проверяет manifest; старые копии доступны только через явный аварийный флаг.
