# Разовый план анализа узких мест

Дата: 21 июля 2026 года  
Статус: план исследования текущей кодовой базы

## Цель и границы

Найти и ранжировать фактические узкие места desktop-приложения: медленные пользовательские операции, дорогие SQL-запросы, избыточные сетевые вызовы, N+1, чрезмерное потребление памяти при работе с файлами и заметные задержки frontend. Результатом анализа должен быть список небольших, измеримо полезных изменений, а не общий рефакторинг.

Уже выполненные оптимизации считать baseline и не повторять без регрессии:

- единый scope доступа основных списков документов;
- batch-загрузка карточек графа и получателей ознакомлений;
- индекс `documents(kind, nomenclature_id, created_at DESC)`;
- cursor pagination основных журналов документов.

## Артефакты анализа

Для каждой найденной проблемы создать запись в рабочем отчёте с полями:

| Поле | Что фиксировать |
|---|---|
| Операция | Действие пользователя и экран |
| Трасса | Wails service → service → repository/storage → frontend consumer |
| Baseline | p50/p95 времени, объём данных, число SQL/HTTP вызовов, память при необходимости |
| Доказательство | SQL plan, профиль, лог запросов либо воспроизводимый сценарий |
| Причина | Например N+1, лишний count, отсутствие индекса, waterfall, большой bundle |
| Варианты | Не менее одного безопасного и обратимого варианта |
| Риск | Доступ, согласованность, UX, write overhead, миграция, rollback |
| Критерий успеха | Конкретная целевая метрика и отсутствие функциональной регрессии |

Не считать проблемой только наличие `Seq Scan`, большого файла или длинной функции: сначала требуется воспроизводимое измерение на объёме, близком к production.

## Шаг 1. Зафиксировать среду и baseline

1. Записать текущий commit, версию PostgreSQL, конфигурацию приложения и параметры тестового набора.
2. Выполнить базовые проверки:

   ```bash
   go test ./...
   make integration-test
   make frontend-build
   make db-performance-check PERFORMANCE_DOCUMENTS=10000
   ```

3. Сохранить результаты `build/performance/latest.json` и summary вне Git либо в задаче/отчёте.
4. Повторить DB benchmark с 50 000–100 000 документов, если production-объём существенно выше 10 000:

   ```bash
   make db-performance-check PERFORMANCE_DOCUMENTS=50000 PERFORMANCE_DEEP_PAGE=1000
   ```

5. Для каждого замера фиксировать warm/cold состояние отдельно: одиночный запуск не считать выводом.

## Шаг 2. Составить карту пользовательских операций

Пройти интерфейс под ролями администратора, регистратора и ограниченного участника. Для каждой операции записать входные параметры, ожидаемый результат и затронутые сервисы.

Приоритетный маршрут:

1. Открытие приложения, восстановление сессии и получение текущего пользователя.
2. Открытие каждого журнала: входящие, исходящие, обращения, приказы.
3. Поиск, фильтрация, переход вперёд/назад по журналу, сброс фильтров.
4. Открытие карточки каждого вида документа, вкладок, вложений, истории и связей.
5. Регистрация и изменение документа, включая связанные поручения и ознакомления.
6. Dashboard, статистика и тяжёлые графики.
7. Задания, замещения, уведомления и административный аудит.
8. Загрузка, скачивание и удаление крупных вложений.
9. Экспорт, backup/restore и outbox-операции, если они используются в эксплуатации.

Для каждого маршрута отметить ожидаемую частоту и вероятный объём данных. Частая операция на 200 ms обычно важнее редкой операции на 2 s.

## Шаг 3. Собрать фактическую latency и число вызовов

1. Временно включить структурированные debug-логи либо локальную обёртку вокруг Wails-bound services с operation ID и длительностью.
2. Для каждого operation ID зафиксировать:
   - общее время Wails-вызова;
   - время SQL и число запросов;
   - обращения к MinIO/HTTP и их длительность;
   - число возвращённых строк, документов и байтов.
3. В frontend использовать browser performance tools:
   - Network/Performance для waterfall и повторных Wails-вызовов;
   - React DevTools Profiler для лишних render и дорогих таблиц;
   - Chrome memory snapshot при открытии/закрытии вложений и модалок.
4. Повторить каждую важную операцию не менее 10 раз и зафиксировать p50 и p95. Не оптимизировать единичный выброс без причины.
5. Отдельно искать последовательные независимые запросы: данные dashboard, вкладки карточки, справочники и параллельные запросы формы.

## Шаг 4. Провести SQL-аудит

### 4.1 Инвентаризация

1. Собрать все SQL-вызовы через `rg -n "Query(Row|Context)?|Exec(Context)?|Prepare" internal`.
2. Сгруппировать их по repository и пользовательской операции из шага 2.
3. Отдельно отметить запросы с:
   - `ILIKE '%…%'`, `OR`, `EXISTS`, `DISTINCT`, `UNION`, CTE и рекурсией;
   - `COUNT(*)`, `OFFSET`, сортировками и широкими `SELECT`;
   - циклами, внутри которых есть запросы;
   - join к таблицам вложений, получателей, резолюций, связей и поручений.

### 4.2 Планы и статистика PostgreSQL

1. На безопасной production-like базе включить `pg_stat_statements`, если это допускает эксплуатационная конфигурация.
2. Снять top SQL по total time, mean time, calls, rows и shared blocks.
3. Для кандидатов выполнить `EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON)` с реальными фильтрами:
   - пустой поиск;
   - распространённый поисковый запрос;
   - редкий поисковый запрос;
   - широкая и ограниченная область доступа;
   - первая и глубокая страница;
   - пустой результат.
4. Сопоставить plan с фактическими индексами, cardinality и объёмом таблиц (`pg_stat_user_tables`, `pg_indexes`).
5. Проверить write cost каждого предлагаемого индекса: вставка/обновление документов, размер индекса, время миграции и down migration.

### 4.3 Специальные проверки

- Проверить рекурсивный CTE графа связей на плотном графе, циклах и ограничении числа узлов.
- Проверить `COUNT(*)` на экранах, где UI не показывает total или может жить с approximate/отложенным числом.
- Проверить поиск по подстроке отдельно от поиска по словам; не добавлять trigram/FTS без подтверждённого набора запросов.
- Проверить, что список возвращает только поля, необходимые для таблицы; тяжёлые детали должны загружаться карточкой.

## Шаг 5. Найти N+1 и лишние round trips

1. Для каждой операции из шага 2 сопоставить количество сущностей и количество SQL/HTTP вызовов.
2. Считать подозрительным рост числа вызовов вместе с числом строк, документов, связей, вложений, получателей или исполнителей.
3. Проверить особенно внимательно:
   - карточки документов и дополнительные вкладки;
   - граф связей;
   - ознакомления и замещения;
   - поручения и соисполнителей;
   - вложения и их метаданные;
   - справочники в формах и селектах.
4. Для подтверждённого N+1 сравнить batch query (`WHERE id = ANY($1)`) с join/preload. Выбрать вариант, который не раздувает число строк и сохраняет границы доступа.
5. Добавить тест, проверяющий число обращений или хотя бы batch-семантику, до внедрения исправления.

## Шаг 6. Проверить backend CPU, блокировки и память

1. На воспроизводимом сценарии снять CPU и heap profile Go (`pprof`) для service/repository/storage операции.
2. Проверить длинные критические секции: транзакции не должны включать тяжёлую сериализацию, передачу файла или внешние HTTP-вызовы без необходимости.
3. Проверить контекстную отмену и timeout для SQL/MinIO вызовов, особенно в действиях UI, которые пользователь может отменить или повторить.
4. Пройти обработку вложений на файлах 10 MiB, 100 MiB и целевом максимальном размере:
   - peak heap;
   - временные файлы;
   - закрытие readers;
   - корректность отмены и cleanup.
5. Проверить worker outbox: размер batch, retry storm, конкуренция с интерактивными запросами и накопление очереди.

## Шаг 7. Проверить frontend, bundle и render

1. Выполнить `make frontend-build` и записать размеры entry/chunk, особенно Statistics, table, graph и модальные окна.
2. Сопоставить большой chunk с route: загружается ли он до того, как пользователь открыл соответствующий экран.
3. В React Profiler пройти таблицы на 10, 50 и 100 строках, поиск с быстрым вводом, открытие карточки и графа.
4. Найти:
   - компоненты, объявленные внутри render;
   - нестабильные props/callbacks, из-за которых перерисовывается таблица;
   - дублирующие `useEffect`-запросы;
   - отсутствие отмены/игнорирования устаревшего ответа;
   - повторную загрузку неизменяемых справочников.
5. Перед внедрением cache, debounce или `Promise.all` проверить семантику ошибок, invalidation после mutation и ограничение параллелизма.

## Шаг 8. Ранжировать кандидаты

Оценить каждый подтверждённый кандидат по шкале 1–5:

| Критерий | Вопрос |
|---|---|
| Пользовательский эффект | Сколько пользователей и как часто сталкиваются с задержкой? |
| Экономия | Насколько уменьшаются p95, SQL calls, blocks, CPU, память или сеть? |
| Уверенность | Есть ли baseline и понятная первопричина? |
| Риск | Может ли изменение нарушить доступ, корректность, UX или восстановление? |
| Стоимость | Размер изменения, миграция, тесты и rollout |

В первую очередь брать задачи с доказанным эффектом, низким риском и локальным изменением. Не объединять в один PR индекс, изменение семантики поиска и крупный frontend refactor.

## Шаг 9. Подготовить изменение для каждого выбранного кандидата

1. Описать точный contract: входные данные, сортировка, access scope, пустой результат, ошибки и совместимость API.
2. Добавить или расширить тест, который воспроизводит исходную проблему.
3. Реализовать минимальное изменение.
4. Для миграции добавить `up`/`down`, тест embedding и оценку lock/write impact.
5. Для frontend-изменения проверить loading, error, race condition, invalidation и доступность клавиатурной навигации.
6. Не менять одновременно поведение и алгоритм без отдельного обоснования.

## Шаг 10. Подтвердить эффект и принять решение

1. Повторить тот же сценарий и объём данных, что в baseline.
2. Сравнить p50/p95, SQL plans, shared blocks, число вызовов, память и размер bundle только для относящихся метрик.
3. Проверить негативные сценарии: пустой набор, широкий поиск, ограниченный доступ, равные timestamps, параллельное изменение данных, крупный файл.
4. Запустить минимум:

   ```bash
   go test ./...
   make integration-test
   make frontend-build
   git diff --check
   ```

5. Для SQL-изменений дополнительно выполнить `make db-performance-check` и сохранить before/after summary.
6. Принять изменение только если измеримый выигрыш превышает сложность и нет регрессии в функциональности, доступе и write path.

## Итоговый порядок выполнения

1. Baseline и карта операций.
2. Логи latency/round trips и frontend profiling.
3. SQL inventory, `pg_stat_statements` и plans.
4. N+1 и file/memory audit.
5. Ранжирование подтверждённых кандидатов.
6. Реализация по одному кандидату с тестами и before/after evidence.
7. Финальный отчёт: выполненные улучшения, отложенные гипотезы, измерения и риски.
