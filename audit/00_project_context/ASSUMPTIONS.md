# Assumptions

Дата аудита: 2026-05-07

## Предположения Этапа A

1. Статус: `changed`
   Production candidate — это исходное состояние кода приложения на момент начала аудита: чистый Git working tree до добавления audit-артефактов, успешный frontend build, Go build и Go tests.

2. Статус: `confirmed`
   Основная production-платформа включает Linux `amd64` и Windows `amd64`. macOS не является production release target на текущем этапе, несмотря на наличие Wails build assets.

3. Статус: `changed`
   PostgreSQL, MinIO и Seq являются обязательными production-сервисами. Отключаемость Seq допускается только для dev/local окружения или аварийной диагностики, но не как штатный production-режим.

4. Статус: `changed`
   Название организации в сайдбаре должно быть настраиваемым и браться из системной настройки `organization_short_name`, а не быть захардкоженным текстом. Реализованный контракт чтения для общего layout: `SettingsService.GetOrganizationShortName()`, доступный всем авторизованным пользователям; настройки не считаются секретом для чтения, но изменять их может только пользователь с системным правом `admin`.

5. Статус: `confirmed`
   Автоматическая нумерация регистрационных номеров должна быть строго без пропусков. Получение и инкремент `nomenclature.next_number` должны быть атомарной частью успешной регистрации документа. Регистрация документа должна поддерживать backend `idempotency_key`, чтобы повторный запрос возвращал уже созданный документ без дубля и повторного инкремента номера.

6. Статус: `changed`
   `config.example.json`, `.envExample` and `docker-compose.yaml` are local/dev examples and are explicitly marked as not production defaults. Production template/guide базируется на утвержденной документации за пределами проекта; ссылку на нее нельзя добавлять в репозиторий или audit context, поэтому проверки B/E/H должны сверяться с ней вручную.

7. Статус: `confirmed`
   Backup/restore scripts считаются операционной частью production/ops контура приложения, так как они работают с PostgreSQL+MinIO и используются для обслуживания данных.

8. Статус: `changed`
   Профили `admin`, `clerk`, `executor`, `mixed` — наследие первых версий, где они были бизнес-ролями. После рефакторинга они являются только UX-классификацией поверх permission model; источником прав остаются системные и document-domain разрешения.

9. Статус: `confirmed`
   Бинарники Linux и Windows были пересобраны 2026-05-07. Текущие размеры свежих артефактов: `build/bin/docflow` — 18,324,384 bytes (около 18 MB), `build/bin/docflow.exe` — 19,566,080 bytes (около 19 MB). Для финального release-аудита размер все равно нужно перепроверять после финальной сборки.

## Что Подтвердить Вручную

На текущий момент все ручные допущения этапа A подтверждены или перенесены в открытые вопросы следующих этапов.

## Подтверждено Вручную

- Production SLO/performance budget: старт приложения до экрана входа — до 5 секунд; открытие основных списков при обычной пагинации/фильтрах — до 2 секунд; тяжелые отчеты/статистика — до 5 секунд; память desktop-приложения в обычной работе — до 512 MB; размер бинарника — текущие 18-19 MB считаются нормой, warning threshold — 100 MB.
- Backup/restore policy: RPO для PostgreSQL+MinIO — потеря данных не более чем за 1 день; RTO — восстановление за 1-2 дня; retention backup-копий — 15 дней; test restore PostgreSQL+MinIO выполняется вручную перед релизом; проверка backup-архивов выполняется через manual test restore; offsite copy реализован штатно отправкой копии на другой сервер; Seq logs не входят в backup-контур и не бэкапятся.
- Backup/restore script path policy: абсолютный путь к `.env` в `backup_smb_tar.sh` и `restore_smb_tar.sh` допустим из-за особенностей запуска через cron; скрипты не обязаны работать из произвольной рабочей директории, но фактический путь должен совпадать с production cron/deployment path по утвержденной документации.
- Network security policy: PostgreSQL, MinIO и Seq работают в закрытом защищенном LAN-контуре; доступ к ним возможен только из LAN, по открытым каналам связи данные не передаются. Поэтому TLS/SSL для внутренних соединений не является обязательным production-требованием, но LAN-границы контура должны быть подтверждены по утвержденной production-документации.
- Dev examples security policy: слабые примерные пароли и local endpoints in repo examples are допустимы для local/dev шаблона, потому что разработка ведется в закрытом контуре и доступ извне к ресурсам невозможен; эти значения явно помечены как non-production defaults.
- Production config policy: production разворачивается вручную через `config/config.json` по утвержденной production-документации; отдельные repo-конфиги для `dev`, `test`, `staging`, `production` не являются обязательным требованием.
- Expected data volume: ориентировочно не более 1000 документов в год, количество вложений пропорционально документам, не более 20 пользователей, средний размер файла около 3 MB, доступный storage около 1 TB.
- Storage policy: максимальный размер одного вложения в production — 15 MB; разрешенные расширения остаются `.pdf,.doc,.docx,.odt,.xls,.xlsx,.ods`; общий MinIO/storage quota как жесткий лимит пока не задается; monitoring thresholds: warning 80%, critical 90% от доступного storage около 1 TB. При текущем baseline достижение этих порогов в ближайшие годы считается математически маловероятным.
- Migration management policy: запуск миграций и rollback в production через UI разрешен только пользователям с системным правом `admin`; дополнительный confirmation/audit workflow не является обязательным требованием. Безопасность `down`-миграций должна быть отдельно проверена на этапах B/H.
- Log retention/access policy: `document_journal` и `admin_audit_log` в PostgreSQL хранятся весь жизненный цикл проекта и не удаляются на уровне приложения. Экспорт не предполагается, отдельная фильтрация не требуется, изменение возможно только прямым доступом к БД, которого в штатной эксплуатации нет. Seq logs не входят в backup-контур и не являются долговременным источником audit trail.
- Document type policy: document types в ближайшее время не планируется расширять; они остаются фиксированными в коде, а не переводятся в справочник.
- Service account policy: для PostgreSQL, MinIO и Seq допустимо использовать один общий технический аккаунт production-контура; отдельные service accounts по каждому сервису не являются обязательным требованием.
